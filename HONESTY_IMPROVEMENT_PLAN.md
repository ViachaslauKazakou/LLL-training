# План улучшения честности ответов химической модели

## 1) Цель

Повысить "честность" модели: уменьшить уверенные неверные ответы, увеличить долю корректных ответов и корректных отказов ("недостаточно данных"), особенно на расчетных задачах по химии.

## 2) Короткий вывод по стратегии

Для текущего проекта приоритет такой:

1. Качественный датасет + раздельные форматы задач.
2. Честная оценка (отдельный eval-набор и метрики).
3. Verifier (проверка чисел, единиц, стехиометрии) поверх генерации.
4. Reranker как дополнительный слой, но не основной механизм честности.

Почему так: reranker выбирает "самый похожий на правильный" вариант, но не гарантирует математическую корректность. Для задач с расчетами нужен явный этап проверки.

## 3) Текущее состояние (исходная точка)

- Подготовка JSONL уже автоматизирована (auto/tasks/corpus), включая train/valid split.
- На valid модель часто сохраняет формат решения, но допускает ошибки в химии/арифметике.
- Значит главная проблема сейчас не только "форма", а "фактическая корректность".

## 4) Целевая архитектура

```
Промпт -> Generator (LLM) -> N кандидатов -> Verifier -> Final answer
                                  |               |
                                  |               +-- проверка чисел/единиц/баланса
                                  +-- разные варианты решения
```

Вариант без multi-candidate тоже полезен:

1. Generator дает решение.
2. Verifier проверяет решение.
3. Если не прошло проверку: либо повторная генерация, либо честный отказ.

## 5) Датасет: что именно нужно добавить

### 5.1 Типы примеров (минимум 3 класса)

1. `reasoning`:
- Вход: "Задача ..."
- Выход: "Решение ... Ответ ..."
- Нужен для пошаговой логики.

2. `short_answer`:
- Вход: "Задача ..."
- Выход: только "Ответ: ..."
- Нужен для точного финального формата без лишнего текста.

3. `abstain`:
- Вход: задача с нехваткой данных или противоречием.
- Выход: "Недостаточно данных для однозначного ответа".
- Нужен для честности и снижения галлюцинаций.

### 5.2 Структура JSONL (пример)

```json
{"task_type":"reasoning","text":"Задача ...\n\nРешение:\n...\nОтвет: ..."}
{"task_type":"short_answer","text":"Задача ...\n\nОтвет: ..."}
{"task_type":"abstain","text":"Задача ...\n\nОтвет: Недостаточно данных для однозначного ответа."}
```

### 5.3 Рекомендации по объему

Минимально рабочий старт:

- reasoning: 300-500 примеров
- short_answer: 150-300 примеров
- abstain: 80-150 примеров

Пропорции можно держать примерно 55/30/15.

## 6) Eval-набор (обязательно отдельно от train)

### 6.1 Принципы

- Никогда не пересекается с train.
- Содержит все 3 типа примеров.
- Включает "пограничные" случаи: близкие вещества, похожие формулы, ловушки по единицам.

### 6.2 Минимальный размер

- Быстрый контур: 50-80 задач.
- Стабильный контур: 150+ задач.

### 6.3 Метрики честности

1. `Exact Answer Accuracy`:
- точное совпадение финального ответа (или нормализованного значения).

2. `Verifier Pass Rate`:
- доля ответов, прошедших проверку verifier-правилами.

3. `Abstain Precision`:
- насколько корректно модель отказывается именно там, где данных недостаточно.

4. `Overconfidence Error Rate`:
- доля уверенных, но неверных ответов (главная целевая метрика для снижения).

## 7) Verifier: практичный MVP

### 7.1 Что проверять в первой версии

1. Извлечение финального ответа (число + единица).
2. Проверка разумности диапазона (например, отрицательная масса недопустима).
3. Согласованность единиц (г, моль, л при н.у.).
4. Базовые стехиометрические соотношения для типовых реакций.

### 7.2 Пример правил

- Если задача про объем газа при н.у. и в ответе нет единицы объема -> fail.
- Если рассчитана масса и значение < 0 -> fail.
- Если ответ не в формате "Ответ: ..." -> soft fail и перегенерация.

### 7.3 Поведение после fail

1. Перегенерация с подсказкой: "проверь единицы и стехиометрию".
2. Если повторный fail -> честный отказ с объяснением, что вычисление не прошло проверку.

## 8) Роль reranker

Reranker стоит добавлять после verifier MVP.

Плюсы reranker:

- Улучшает выбор среди нескольких кандидатных решений.
- Может повышать качество формулировки и структуру ответа.

Ограничения:

- Не заменяет расчетную проверку.
- Может выбрать "красивый", но математически неверный ответ.

Рекомендуемый порядок:

1. Generator + Verifier.
2. Затем Multi-candidate + Reranker + Verifier.

## 9) План внедрения по этапам

### Этап A (1-2 дня): Датасет и eval

1. Расширить набор с тремя типами (`reasoning`, `short_answer`, `abstain`).
2. Выделить отдельный eval-файл без пересечений.
3. Добавить скрипт проверки пересечений train/eval и базовой статистики.

Результат: измеримая база качества.

### Этап B (1-2 дня): Verifier MVP

1. Реализовать правила проверки финального ответа.
2. Добавить режим "перегенерация при fail".
3. Логировать причину fail по каждой задаче.

Результат: снижение уверенных ошибок.

### Этап C (2-3 дня): Улучшение инференса

1. Self-consistency (3-5 кандидатов).
2. Опциональный reranker.
3. Финальный отбор: только через verifier pass.

Результат: более стабильные ответы на сложных задачах.

## 10) Ожидаемые преимущества

1. Меньше уверенных неверных ответов.
2. Больше корректных финальных ответов в формате "Ответ: ...".
3. Прозрачная диагностика: видно, где ошибка в данных, а где в рассуждении.
4. Контролируемый рост качества через объективные метрики, а не "по ощущениям".

## 11) Критерии успеха (пример)

- Exact Answer Accuracy: +15-25% относительно текущей базы.
- Overconfidence Error Rate: -30% и лучше.
- Abstain Precision: >= 0.85 на абстейн-наборе.
- Доля ответов, прошедших verifier: устойчивый рост от итерации к итерации.

## 12) Практические примеры внедрения

### Пример 1: short_answer формат

Вход:

"Задача: Какой объем H2 (н.у.) выделится при взаимодействии 13 г Zn с избытком HCl?"

Ожидаемый выход:

"Ответ: 4,48 л"

### Пример 2: abstain формат

Вход:

"Задача: Рассчитайте молярную массу вещества X, если известно только, что X - газ."

Ожидаемый выход:

"Ответ: Недостаточно данных для однозначного ответа."

### Пример 3: verifier fail -> retry

1. Модель дала: "Ответ: -5 г".
2. Verifier: fail (отрицательная масса).
3. Retry prompt: "Проверь знак, единицы и стехиометрию."
4. Новый ответ проходит проверку.

## 13) Риски и как их снизить

1. Риск: модель заучит шаблоны отказов.
- Митигировать: разнообразить формулировки abstain-примеров.

2. Риск: verifier слишком жесткий.
- Митигировать: разделить hard fail и soft fail правила.

3. Риск: маленький eval и шумные выводы.
- Митигировать: увеличить eval и вести сравнение на фиксированном benchmark-наборе.

## 14) Что делать прямо сейчас

1. Сформировать v2-датасет с тремя типами примеров.
2. Создать отдельный eval-набор (минимум 50 задач).
3. Добавить verifier MVP для финального ответа и единиц.
4. Повторить fine-tuning и сравнить метрики до/после.
