## Типичная скорость генерации

Скорость генерации текста в языковых моделях измеряется в **tokens/second (токенов в секунду)**.

### Референсные значения для разных размеров моделей:

**На CPU (Apple M1/M2/M3):**
- Tiny модель (3-10M параметров): **50-150 tok/s**
- Small модель (10-50M параметров): **20-80 tok/s**
- Medium модель (50-150M параметров): **5-30 tok/s**
- Large модель (150M+ параметров): **1-10 tok/s**

**На MPS (Apple Silicon GPU):**
- Tiny модель: **100-300 tok/s**
- Small модель: **50-150 tok/s**
- Medium модель: **20-60 tok/s**
- Large модель: **5-20 tok/s**

**На CUDA (NVIDIA GPU, например RTX 3090):**
- Tiny модель: **300-1000 tok/s**
- Small модель: **150-500 tok/s**
- Medium модель: **50-200 tok/s**
- Large модель: **20-80 tok/s**

### Для сравнения — коммерческие модели:

- **GPT-3.5**: ~60-100 tok/s через API
- **GPT-4**: ~20-40 tok/s через API
- **Claude 3**: ~50-80 tok/s через API
- **Llama 2 70B** (локально на A100): ~15-30 tok/s

## Факторы, влияющие на скорость генерации (без учета железа)

### 1. Размер модели (количество параметров)

**Влияние:** Чем больше параметров, тем медленнее генерация.

**Почему:**
- Каждый токен требует вычисления через все слои модели
- Больше параметров = больше матричных умножений
- Больше параметров = больше данных для загрузки из памяти

**Пример:**
```python
# 3M параметров: ~100 tok/s на CPU
config_tiny = ModelConfig(
    vocab_size=5000,
    d_model=128,
    n_layers=4,
    n_heads=4
)

# 13M параметров: ~40 tok/s на CPU
config_small = ModelConfig(
    vocab_size=10000,
    d_model=256,
    n_layers=4,
    n_heads=4
)

# 50M параметров: ~10 tok/s на CPU
config_medium = ModelConfig(
    vocab_size=10000,
    d_model=512,
    n_layers=8,
    n_heads=8
)
```

**Как оптимизировать:**
- Используйте минимально необходимый размер модели для вашей задачи
- Для быстрой генерации выбирайте `d_model=128-256`, `n_layers=4-6`
- Уменьшите `vocab_size` (меньше словарь = быстрее финальный softmax)

---

### 2. Длина контекста (context_len)

**Влияние:** Чем длиннее контекст, тем медленнее генерация.

**Почему:**
- Attention механизм имеет сложность O(n²), где n — длина контекста
- При генерации каждого токена модель должна обработать весь предыдущий контекст
- Чем длиннее промпт + сгенерированный текст, тем медленнее каждый следующий токен

**Пример:**
```python
# context_len=128: быстро
# context_len=512: в 4 раза медленнее
# context_len=2048: в 16 раз медленнее
```

**Как оптимизировать:**
- Используйте минимально необходимую длину контекста
- Для задач с короткими ответами достаточно `context_len=128-256`
- Не используйте `context_len=2048`, если вам нужно генерировать только 50-100 токенов
- Обрезайте промпт до необходимого минимума

---

### 3. Параметры генерации

#### 3.1 Temperature

**Влияние:** Минимальное, но есть.

**Почему:**
- Temperature применяется к логитам перед softmax: `logits / temperature`
- Это простая операция деления, почти не влияет на скорость

**Как оптимизировать:**
- Не влияет на скорость, можно не трогать

#### 3.2 Top-k sampling

**Влияние:** Очень небольшое.

**Почему:**
- Top-k требует сортировки/выбора топ-k токенов из словаря
- Для небольших k (10-100) влияние минимально
- Для k=vocab_size эффект нулевой

**Пример:**
```python
# top_k=10: быстро
# top_k=100: почти так же быстро
# top_k=10000: чуть медленнее (но незначительно)
```

**Как оптимизировать:**
- Не влияет существенно на скорость
- Используйте разумные значения (10-100)

#### 3.3 Batch size при генерации

**Влияние:** При генерации batch_size обычно = 1 (один промпт).

**Почему:**
- Генерация — авторегрессивный процесс (каждый токен зависит от предыдущих)
- Нельзя распараллелить генерацию токенов внутри одной последовательности
- Можно генерировать несколько независимых последовательностей параллельно (batch_size > 1)

**Как оптимизировать:**
- Если нужно сгенерировать несколько вариантов, используйте batch_size > 1
- Это даст прирост скорости через параллелизм

---

### 4. Архитектура модели

#### 4.1 Число голов внимания (n_heads)

**Влияние:** Минимальное при правильной реализации.

**Почему:**
- Multi-head attention эффективно реализован в PyTorch
- Вычисления распараллеливаются
- Важно, чтобы `d_model % n_heads == 0`

**Как оптимизировать:**
- Используйте `n_heads=4` для small моделей
- Используйте `n_heads=8` для medium моделей
- Убедитесь, что `d_model` делится на `n_heads`

#### 4.2 Feed-forward размер (d_ff)

**Влияние:** Существенное.

**Почему:**
- Feed-forward слой обычно имеет размер `d_ff = 4 * d_model`
- Это большая часть вычислений в каждом слое
- Больше d_ff = медленнее

**Как оптимизировать:**
- Используйте `d_ff = 4 * d_model` (стандарт)
- Можно попробовать `d_ff = 2 * d_model` для ускорения (но может снизить качество)

#### 4.3 Тип активации

**Влияние:** Очень небольшое.

**Почему:**
- GELU vs ReLU — разница минимальна
- Современные библиотеки оптимизированы

**Как оптимизировать:**
- Используйте GELU (стандарт для трансформеров)
- Не влияет существенно на скорость

---

### 5. Размер словаря (vocab_size)

**Влияние:** Существенное для финального softmax.

**Почему:**
- При генерации каждого токена нужно вычислить вероятности для всех токенов словаря
- Финальный слой: `(d_model, vocab_size)` — большая матрица
- Softmax по vocab_size токенам

**Пример:**
```python
# vocab_size=1000: быстро
# vocab_size=10000: чуть медленнее
# vocab_size=50000: заметно медленнее
```

**Как оптимизировать:**
- Используйте минимальный словарь для вашей задачи
- CharTokenizer (посимвольный) имеет малый vocab (~100-200 символов) — очень быстро
- WordTokenizer имеет большой vocab (10k-50k слов) — медленнее
- BPE tokenizer — компромисс (vocab ~5k-10k)

---

### 6. Оптимизации модели

#### 6.1 KV-кэширование (Key-Value caching)

**Влияние:** Критическое — ускорение в 2-10 раз!

**Почему:**
- При генерации каждого нового токена не нужно пересчитывать ключи и значения (keys, values) для всех предыдущих токенов
- Можно кэшировать их и переиспользовать

**Наша реализация:** В нашем коде KV-кэш не реализован (простая реализация).

**Как реализовать:**
```python
def generate_with_kv_cache(self, input_ids, max_new_tokens, ...):
    past_key_values = None  # кэш
    
    for _ in range(max_new_tokens):
        # Первый проход: обработать весь промпт
        # Последующие: обработать только последний токен + использовать кэш
        logits, past_key_values = self.forward_with_cache(
            input_ids[:, -1:],  # только последний токен
            past_key_values=past_key_values
        )
        # ...
```

**Ускорение:**
- Для длинных контекстов (512+): **5-10x быстрее**
- Для коротких (128): **2-3x быстрее**

#### 6.2 Flash Attention

**Влияние:** Существенное ускорение (1.5-3x).

**Почему:**
- Оптимизированная реализация attention
- Уменьшает использование памяти
- Быстрее вычисления

**Как использовать:**
```python
# Требует установки flash-attn
import torch.nn.functional as F
outputs = F.scaled_dot_product_attention(
    query, key, value,
    attn_mask=mask,
    is_causal=True
)
```

#### 6.3 Model quantization (квантизация)

**Влияние:** Ускорение в 2-4 раза + экономия памяти.

**Почему:**
- Уменьшение точности весов: float32 → int8 или int4
- Меньше данных для загрузки из памяти
- Быстрее вычисления на некоторых устройствах

**Наша реализация:** Не поддерживается (обычный float32).

**Как реализовать:**
```python
# PyTorch quantization
model_int8 = torch.quantization.quantize_dynamic(
    model, {torch.nn.Linear}, dtype=torch.qint8
)
```

**Ускорение:**
- int8: **2-3x быстрее**, небольшая потеря качества
- int4: **3-4x быстрее**, заметная потеря качества

---

### 7. Оптимизация промпта

**Влияние:** Может существенно ускорить.

**Почему:**
- Короткий промпт = меньше токенов для обработки
- Меньше контекста = быстрее attention

**Как оптимизировать:**
```python
# Плохо: длинный промпт
prompt = """
Ты — опытный программист на Python. 
У тебя большой опыт работы с различными фреймворками.
Пожалуйста, напиши функцию для сортировки списка.
"""

# Хорошо: короткий промпт
prompt = "Функция сортировки списка на Python:"
```

## Типичные проблемы и решения

### Проблема 1: "Генерация очень медленная (< 5 tok/s)"

**Возможные причины:**
1. Слишком большая модель для вашего устройства
2. Длинный контекст (context_len > 512)
3. CPU вместо GPU

**Решения:**
- Уменьшите размер модели (d_model, n_layers)
- Уменьшите context_len до 128-256
- Проверьте, что используется GPU (MPS/CUDA), а не CPU

### Проблема 2: "Скорость падает со временем генерации"

**Причина:** 
- Attention сложность O(n²) — чем больше сгенерировано токенов, тем медленнее следующий

**Решение:**
- Реализовать KV-кэширование
- Ограничить max_new_tokens
- Использовать sliding window attention

### Проблема 3: "Первый токен генерируется медленно"

**Причина:**
- Первый проход обрабатывает весь промпт сразу
- Последующие токены генерируются по одному

**Решение:**
- Это нормально
- Сократить промпт
- Использовать KV-кэш (чтобы не пересчитывать промпт при каждом токене)

## Измерение и мониторинг скорости

### Метрики для отслеживания:

1. **tokens/second** — основная метрика скорости
2. **time_to_first_token (TTFT)** — задержка до первого токена
3. **time_per_token** — среднее время на токен (после первого)
4. **throughput** — общая пропускная способность

### В нашем коде:

```python
stats = {
    'total_time': total_time,              # общее время (сек)
    'tokens_generated': tokens_generated,  # количество токенов
    'tokens_per_second': tokens_per_second,# скорость (tok/s)
    'prompt_tokens': prompt_tokens         # токенов в промпте
}
```

### Как интерпретировать:

- **> 50 tok/s** — отлично для CPU
- **20-50 tok/s** — хорошо для CPU
- **10-20 tok/s** — приемлемо
- **< 10 tok/s** — медленно, нужна оптимизация
